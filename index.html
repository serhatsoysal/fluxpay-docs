API Reference

Complete REST API documentation for FluxPay. All endpoints require JWT authentication and support multi-tenant operations.

Authentication

All API requests require a valid JWT access token in the Authorization header: 

Authorization: Bearer <access_token>

For multi-tenant operations, include the tenant ID in the header: 

X-Tenant-ID: <tenant_id>

Base URL

All API endpoints are relative to the base URL: 

https://fluxpay-api.onrender.com/api

Additional Information

Pagination

Paginated endpoints use: page (zero-based), size, content, totalElements, totalPages

Currency

All monetary amounts are stored as integers representing the smallest currency unit (e.g., cents for USD). $29.99 is represented as 2999.

Rate Limiting

API requests are rate-limited per tenant. Rate limit headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset

Error Responses

All error responses include: timestamp, status, error, message, path

Common status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 500 Internal Server Error

What is FluxPay?

FluxPay is a subscription management and billing orchestration platform. It helps businesses manage recurring payments, subscriptions, invoices, and customer relationships in one place.

Why FluxPay?

Managing subscriptions and billing can be complex. FluxPay simplifies this with:

  • •Automated billing and invoicing
  • •Multi-tenant architecture
  • •Real-time analytics
  • •Secure payment processing

How Does It Work?

Built with modern technologies:

  • •Backend: Java 21 + Spring Boot
  • •Frontend: React 18.2 + TypeScript
  • •Database: PostgreSQL
  • •Security: JWT authentication

Where Can You Use It?

Perfect for SaaS companies, e-commerce platforms, digital service providers, and any business that needs recurring payment processing.

When to Use It?

Use FluxPay when you need recurring payment management, automated invoicing, multi-tenant billing, or real-time subscription analytics.

Designed for developers, business owners, product teams, and finance teams who need a scalable, secure subscription management solution.

Module Structure

FluxPay consists of 8 independent modules, each with specific responsibilities following Domain-Driven Design principles.

1. fluxpay-common

Purpose: Shared utilities, base entities, DTOs, enums, and exceptions

Key Components:

  • • BaseEntity: Abstract base class with common fields
  • • DTOs: Shared data transfer objects
  • • Enums: Common enumerations
  • • Exceptions: Custom exception hierarchy

2. fluxpay-security

Purpose: Authentication, authorization, session management, and tenant context

Key Components:

  • • JwtTokenProvider: JWT token creation and validation
  • • SessionService: Enterprise session management
  • • TenantContext: ThreadLocal-based tenant isolation
  • • RateLimitService: Request rate limiting

3. fluxpay-tenant

Purpose: Tenant and user management

Key Components:

  • • Tenant: Tenant entity
  • • User: User entity with roles
  • • ApiKey: API key management
  • • TenantService: Tenant business logic

4. fluxpay-product

Purpose: Product catalog and pricing models

Key Components:

  • • Product: Product entity
  • • Price: Pricing models (flat-rate, per-unit, tiered)
  • • Feature: Product features
  • • ProductService: Product management

5. fluxpay-subscription

Purpose: Subscription lifecycle management

Key Components:

  • • Subscription: Subscription entity with lifecycle states
  • • SubscriptionItem: Subscription line items
  • • Customer: Customer entity
  • • SubscriptionService: Subscription lifecycle management

6. fluxpay-billing

Purpose: Invoice generation and payment processing

Key Components:

  • • Invoice: Invoice entity
  • • Payment: Payment transactions
  • • InvoiceService: Invoice generation and management
  • • PaymentService: Payment processing

7. fluxpay-notification

Purpose: Email and webhook notifications

Key Components:

  • • EmailTemplate: Email template management
  • • WebhookEndpoint: Webhook endpoint configuration
  • • EmailService: Email sending service
  • • WebhookService: Webhook delivery service

8. fluxpay-api

Purpose: REST API layer, controllers, and main application

Key Components:

  • • AuthController: Authentication endpoints
  • • ProductController: Product endpoints
  • • SubscriptionController: Subscription endpoints
  • • PaymentController: Payment endpoints

Security Features

Password Hashing

BCrypt algorithm with automatic salt generation (10 rounds default)

Token Blacklisting

Invalidated tokens stored in Redis with TTL matching token expiration

Rate Limiting

Redis-based rate limiting: 5 session creations/min per IP, 1000 API requests/min per session

Device Fingerprinting

ua-parser library for device tracking and anomaly detection

Concurrent Session Control

Maximum 5 sessions per user with oldest session eviction strategy

Anomaly Detection

Tracks suspicious activity, failed attempts, and geographic anomalies

Architecture & Design

The project follows a Feature-Based Modular Architecture with clear separation of concerns, utilizing modern design patterns and component composition.

Architectural Approach

  • •Feature Modules: Each business feature is self-contained
  • •Shared Layer: Reusable components, utilities, and types
  • •Application Layer: Root configuration, providers, and routing

Design Patterns

1. Feature-Based Organization

Each feature module contains: api/, components/, hooks/, pages/, types/, utils/

2. Container/Presentational Pattern

Container components handle data fetching; presentational components focus on UI rendering

3. Custom Hooks Pattern

Business logic abstracted into reusable hooks (useAuth, useSubscriptions, useDebounce)

4. Provider Pattern

QueryClientProvider and AppProviders wrap application with necessary contexts

5. Compound Components Pattern

Radix UI components use compound component pattern for flexible composition

State Management

Hybrid state management approach with three distinct layers: Global State (Zustand), Server State (React Query), and Local State (React Hooks).

State Management Layers

1. Global State (Zustand)

Purpose: Authentication and user context

Features:

  • • Auth store with user info and tenant ID
  • • Cross-tab synchronization via BroadcastChannel
  • • Token management and localStorage persistence
  • • Authentication state initialization

2. Server State (React Query)

Purpose: API data, caching, and synchronization

Features:

  • • Automatic caching and background refetching
  • • Optimistic updates for better UX
  • • Query invalidation and cache management
  • • Error handling and retry logic

3. Local State (React Hooks)

Purpose: Component-specific UI state

Features:

  • • UI state (modals, dropdowns, toggles)
  • • Form state (React Hook Form)
  • • Temporary calculations
  • • Component-specific preferences

Request Interceptor

apiClient.interceptors.request.use((config) => {
  // Add Tenant ID header
  const tenantId = useAuthStore.getState().tenantId;
  if (tenantId) {
    config.headers['X-Tenant-ID'] = tenantId;
  }
  
  // Add Authorization token
  const token = tokenManager.getToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  
  return config;
});

Response Interceptor

Handles token refresh on 401 errors:

  • •Detects 401 Unauthorized responses
  • •Queues failed requests during token refresh
  • •Automatically retries original request after refresh
  • •Logs out user if refresh fails
